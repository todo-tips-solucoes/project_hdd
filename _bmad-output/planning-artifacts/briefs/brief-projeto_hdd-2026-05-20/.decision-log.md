## 2026-05-20 — Nova síntese do brief (proposta, pendente confirmação)

O brief documenta **um ambiente operacional** com 5 componentes:

1. **Discovery + Planning interativos** (Fases 1-2 BMAD): operador dirige; agentes assistem (`bmad-product-brief`, `bmad-prd`).
2. **Execução autônoma** (Fases 3-4 BMAD): sub-agentes rodam arquitetura, épicos, código, testes, revisão sem supervisão contínua.
3. **Critérios de interrupt:** lista declarada de gatilhos que disparam mensagem WhatsApp ao operador (a definir — pergunta crítica #2).
4. **Canal de interrupt:** WhatsApp bidirecional via solução já existente do usuário (a esclarecer — pergunta crítica #1).
5. **Gates de qualidade entre handoffs** (PM→Architect, Architect→Dev, Dev→QA): cada passagem só ocorre se checklist atendido.

## 2026-05-20 — Respostas a 3 perguntas críticas

### Q1 — Stack WhatsApp
- Provedor: **sistema proprietário do operador** (não Z-API/Twilio/Cloud API).
- Bidirecional: **sim**.
- Hosting: **VPS própria**.
- Número aprovado: **sim, operacional**.
- Implicação: controle total da stack, sem dependência de provedor externo. Risco residual (banimento Meta para soluções não-oficiais) documentado no addendum como item a decidir antes de v1.

### Q2 — Critérios de decisão crítica
- **Confirmado:** Discovery + PRD + Arquitetura são SEMPRE colaborativos (operador + agentes Claude Code interativos). NÃO entram em modo autônomo.
- **Trigger primário e único:** o agente Reviewer (`bmad-code-review`) detecta uma decisão de produto ou arquitetura que não foi prevista no PRD/Arquitetura e dispara WhatsApp.
- Gatilhos secundários (falha reincidente, blocker técnico, custo, conflito, watchdog) reconhecidos pelo usuário ("todo que falou faz sentido") e formalizados no addendum, mas em escopo de **arquitetura**, não brief.

### Q3 — OpenClaw vs. nativo
- **Híbrido (Opção C escolhida):** Discovery/Planning no Claude Code interativo (BMAD nativo). Execução autônoma Fases 3-4 num worker OpenClaw em VPS própria.
- Implica instalar o plugin `ErwanLorteau/BMAD_Openclaw` no worker.
- Workspace canônico segue em `/var/lib/projeto_hdd/_bmad-output/` local; sincronização worker↔local pendente decisão (arquitetura).
