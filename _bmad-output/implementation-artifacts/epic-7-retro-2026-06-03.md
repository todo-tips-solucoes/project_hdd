## Resumo do épico

Origem: decisão de direção 2026-06-02 (`epic-7-scope-proposal.md`), não o PRD original. Objetivo:
validar o HDD **usando-o para construir features reais**, de forma faseada e instrumentada, e
expor gaps reais. Hipótese primária **H-A (Capacidade)**: *"o HDD leva features de ponta a ponta
com o humano só nos gates."*

**Entregue:**
- **Bloco A (harness):** 7.1 métricas de dogfood; 7.2 loop gaps→backlog; 7.3 repo-alvo (`paulotodo/hdd-calibragem`).
- **Fase 1 (calibração):** 7.4 `cep` (baseline) · 7.5 `cnpj` + `data_br`. **3/3 features reais one-shot** (reached_gate, 0 correções, 0 escaladas, 0 leases vazados), pipeline `clone→claude→verify→PR→gate→merge` validado ponta a ponta.
- **Gate Fase 2:** 7.6 (NO-GO condicional) → 7.7 (hardening PC-1) → gate re-rodado → **GO**.
- **Planejada:** 7.8 — 1ª meta-onda (verify com oracle oculto).

**Incidentes:** 1 (OOM por composição, 2026-06-02 — mitigado e formalizado). **ADRs gerados:** 0005, 0006. **Produção:** intacta o tempo todo.

## O que funcionou (manter)

1. **Gate verificável > convenção.** O padrão "salvaguarda nasce como gate em código/teste" se repetiu e pagou: pré-flight de capacidade (OOM), invariante de mounts (PC-1), critério de GO ancorado em métricas. Virou DoD.
2. **Gate de fundação como modelo de decisão.** O gate 7.6 fez exatamente seu papel — provou capacidade **e** barrou a auto-modificação no prod até a contenção endurecer. NO-GO foi um sucesso do processo, não uma falha.
3. **Faseamento (separado → meta).** Calibrar num repo de baixo risco antes de tocar o próprio HDD expôs os gaps reais (OOM, oracle visível, PC-1) **sem** risco de auto-modificação. Validou o desenho do épico.
4. **Correct-course disciplinado.** O OOM virou ADR + pré-flight + DoD em vez de "ajusta e segue". Evidência preservada (ondas mortas, dumps).
5. **Honestidade sobre métricas.** Quando o `subscription` se mostrou incapaz de emitir tokens, a métrica D-032 foi **reformulada** (pressão de quota observável) em vez de fingida.

## O que surpreendeu (gaps reais expostos — o objetivo do épico)

1. **OOM por composição (não bug).** A VPS de 8 GB sem swap afogou sob prod + dev + driver-no-host. **Lição:** capacidade do host é uma salvaguarda de 1ª classe; dogfood/calibração competem por RAM com o que medem. → pré-flight + VPS dedicada de 16 GB (ADR 0005).
2. **Oracle visível → one-shot.** O agente do `execute` lê os `tests/test_*.py` no clone, então modelos capazes (até haiku) implementam direto para os casos visíveis. **O loop de correção não dispara naturalmente.** Forçá-lo com alvo adversarial falhou 2×. → AC redefinida (correção é medida, não forçada) + backlog "oracle oculto" (Story 7.8).
3. **PC-1 era do caminho host, não da Fase 2.** O risco de Write fora do workspace existe no `calibration_wave.py` (host), mas **não** na Fase 2: o `execute` roda no container worker, que não monta a árvore de prod/secrets. Contenção por construção — fechável **sem quota**, só entendendo o sistema. **Lição:** mapear o caminho de execução real antes de assumir o risco.
4. **`subscription` não mede custo.** O driver `claude -p` não emite tokens/custo; o `quota_lease` conta concorrência interna, não uso da conta. Medição real de quota só com o driver `api` (Epic 8).

## Lições para a Fase 2 (meta-ondas) e o Epic 8

- **Calibração on-host é Fase-1-only.** Meta-ondas **devem** rodar in-container (produtor→worker). Documentado no ADR 0006 e no `factory.py`.
- **Oracle oculto** (Story 7.8) é o que destrava medir o loop de correção de verdade — e é a 1ª meta-onda (o HDD melhora o próprio harness).
- **Driver `api`** (Epic 8) para medir tokens/custo reais — pré-requisito de qualquer afirmação sobre viabilidade de quota (H-C).
- **Primeira meta-onda = baixo risco e bem-delimitada** (sabedoria reforçada): MVP retrocompatível, gate humano reforçado.

## Atualização — Fase 2 (meta-dogfood) CONCLUÍDA (2026-06-03, sessão da meta-onda 1)

> A retro acima foi escrita com a Fase 2 à frente. Esta seção fecha o ciclo: o HDD se
> auto-modificou num alvo real. Equivale à Story 7.10 do escopo (retro de dogfood + direção
> do Epic 8). Decisão do operador na facilitação: **seguir com mais meta-ondas, em sessão limpa.**

### O que entrou desde a retro de Fase 1

- **7.7 hardening (PC-1)** — contenção do `execute` pelo boundary do worker container, pinada
  por invariante executável. Fechou o NO-GO condicional → **GO** (ADR 0006).
- **7.8 meta-sandbox** — imagem `hdd-meta-sandbox:latest` (Python 3.13 + deps do backend) roda
  a suíte do próprio `projeto_hdd` no `verify` (`--network none`). Infra de bootstrap montada
  **direto pelo dev** (a meta-onda não pode construir o que ela própria precisa para verificar).
- **7.9 meta-onda 1 — oracle oculto** — **primeira auto-modificação real**: o HDD construiu
  `oracle_dir` opcional no `SandboxConfig` + mount `-v <oracle>:/oracle:ro` só no `verify`,
  retrocompatível, com testes. In-container, **one-shot** (`reached_gate`, 0 correções, 0
  escaladas, 0 leases vazados). Gate humano = GO → **PR #27 merged** (`fc8d2b2`).

### O que funcionou na Fase 2 (manter)

1. **Pipeline de auto-modificação validado ponta a ponta.** `hdd start` → worker container →
   execute no clone efêmero → verify no meta-sandbox → PR → gate humano. O HDD melhorou o
   próprio harness, com a produção no ar e intacta.
2. **Boundaries seguraram na prática (não em teoria).** PC-1 (execute contido pelo worker) e
   PC-2 (sem auto-deploy) exercitados ao vivo. `compose.meta.yaml` espelhou o boundary de prod
   sem regressão. Segurança **por construção**.
3. **Gate humano > verify automático.** O `verify` só roda `pytest`; a rede de segurança real
   foi a revisão DoD completa no gate (ruff + mypy --strict + import-linter), que pegou o que o
   verify não pega (boundary/tipo). Reforça o gate como camada de qualidade, não burocracia.
4. **Infra reproduzível como artefato.** `compose.meta.yaml` (committado) torna a próxima
   meta-onda um `up -d` — não mais uma montagem manual. Token fora do `docker inspect` (auth
   pelo credential helper do `gh`).

### Gaps novos expostos pela meta-onda (honestos)

1. **Recursão tem piso de bootstrap.** O HDD não se auto-modifica do zero — precisou do
   meta-sandbox montado à mão. **Lição:** a auto-melhoria tem um degrau que o humano coloca.
2. **PR nasce _draft_.** O nó de PR abre draft; o gate marca "ready" antes do merge (reforça
   "sem auto-merge"). Documentar como comportamento esperado.
3. **Meta-sandbox carrega deps da build.** Onda que adicione dependência nova quebra o `verify`
   até rebuild da imagem. Candidato a automação (rebuild no `uv.lock` mudado).
4. **Merge ≠ deploy.** O código entrou na `main`; o worker de **produção** só usa a feature
   após rebuild+restart (runbook). Auto-modificação **não** é auto-deploy — e isso é proposital.
